# Block Snap

**Minecraft 实例变更追踪与游戏资产知识聚合平台**

---

## 项目概述

Block Snap 是一套面向 Minecraft Java Edition 玩家与整合包维护者的**实例状态变更追踪系统**。它通过客户端模组在每次游戏启动时自动采集实例内全部游戏资产的完整快照，经由后端服务持久化、差分分析与外部平台信息聚合，最终在 Web 端呈现**跨时间维度的资产变更全景**。

项目不替代启动器或模组平台的安装/更新能力，而是填补现有工具链在**历史状态追踪**与**变更认知**方面的空白——帮助用户回答：

> 在一段时间没有游玩之后，我的实例里哪些资产发生了变化？这些变化彼此如何关联？对兼容性和游戏体验有何影响？我现在应该如何处理？

---

## 背景与问题

Minecraft 游戏实例由五类异构资产共同构成：

| 资产类别 | 说明 | 典型变更特征 |
|----------|------|--------------|
| 模组（Mods） | Forge/Fabric 加载的 `.jar` 扩展 | 更新频繁，版本间 API 变更可能导致不兼容 |
| 整合包（Modpacks） | CurseForge/Modrinth 清单或启动器导出 | 模组组合与版本整体漂移 |
| 资源包（Resource Packs） | 纹理、音效、语言等 `.zip` 资源 | Pack Format 版本与 MC 版本强绑定 |
| 光影包（Shader Packs） | OptiFine/Iris 下的 GLSL 着色器 | 依赖渲染管线与 GPU 环境 |
| 配置文件（Configs） | `config/` 目录下各模组配置 | 玩家手动修改多，模组升级后格式易失效 |

上述资产体系存在四类结构性痛点：

1. **多维更新耦合** — 模组版本变更常联动配置文件格式、资源包兼容性与整合包声明，变更并非孤立事件。
2. **信息高度碎片化** — 更新日志分散于 CurseForge、Modrinth、GitHub、Planet Minecraft 等平台，缺乏统一入口。
3. **时间隔阂效应** — 玩家间歇性游玩，回归时实例已在后台发生大量变更，信息严重滞后。
4. **配置漂移** — 配置文件缺乏版本控制，模组升级后难以判断是否需要重置或手动迁移。

Prism Launcher、CurseForge App、Modrinth App 等现有工具聚焦**当前时刻**的资产管理（安装、启用、更新），不持久化历史状态，不提供跨资产、跨时间的变更分析能力。

## 解决方案

Block Snap 由三层子系统协作完成「采集 → 处理 → 呈现」闭环：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐     ┌──────────────┐
│ 客户端 Mod  │────▶│   RabbitMQ   │────▶│  Spring Cloud   │────▶│  Web 仪表盘  │
│  五类资产    │     │  异步上报     │     │  快照/Diff/聚合  │     │  变更可视化   │
│  全量扫描    │     │              │     │                 │     │              │
└─────────────┘     └──────────────┘     └────────┬────────┘     └──────────────┘
                                                  │
                     ┌────────────────────────────┼────────────────────────────┐
                     ▼                            ▼                            ▼
               ┌──────────┐              ┌──────────────┐              ┌──────────┐
               │  MySQL   │              │ Elasticsearch │              │  Redis   │
               │ 关系存储  │              │ 全文检索/聚合  │              │ 缓存/会话 │
               └──────────┘              └──────────────┘              └──────────┘
                                                  ▲
                     ┌────────────────────────────┘
                     │  CurseForge / Modrinth / GitHub（定时采集）
                     └──────────────────────────────────────────────
```

### 客户端采集层

Fabric/Forge 模组在游戏加载生命周期中监听启动事件，对实例目录进行全量扫描，采集结果聚合为一次**实例快照**后异步上报：

| 资产 | 扫描路径 | 核心采集字段 |
|------|----------|--------------|
| 模组 | `mods/*.jar` | modId、版本、SHA-256、修改时间、加载耗时 |
| 整合包 | 清单文件 | 名称、版本、所含模组列表 |
| 资源包 | `resourcepacks/*.zip` | Pack Format、哈希、修改时间 |
| 光影包 | `shaderpacks/*.zip` | 名称、哈希、修改时间 |
| 配置文件 | `config/**` | 相对路径、哈希、关联模组推断 |

采集过程异步执行，不阻塞游戏主线程。

### 服务端处理层

基于 Spring Cloud 微服务架构，承担以下职责：

- **快照消费与 Diff** — 对比历史快照，按资产类别计算新增、移除、版本变更、哈希变更事件
- **平台信息聚合** — 定时从 CurseForge、Modrinth 等平台采集版本更新日志，识别破坏性变更
- **关联分析** — 将平台数据与实例变更匹配，分析配置文件漂移与模组版本升级的联动关系
- **风险评估** — 对变更进行兼容性风险分级（安全 / 需关注 / 破坏性变更）

### Web 展示层

单页应用提供面向用户的数据视图，核心模块包括：

- **我的实例** — 实例列表与资产概要、更新提示、配置漂移告警
- **变更时间线** — 选择实例，对比两次快照，展示五类资产的 ADD / REMOVE / UPDATE Diff
- **版本更新** — 聚合平台更新信息流，标记破坏性变更
- **MC 大事件** — 影响生态的重要事件时间线（版本发布、安全漏洞、模组里程碑等）
- **诊断分享** — 生成可公开访问的实例快照链接，便于协作排查

## 核心数据模型

系统以**实例（Instance）**为中心组织数据，而非单次启动记录：

```
Account（账户）
  └── Instance（游戏实例）
        └── Snapshot[]（历次快照）
              └── Assets（五类资产）
                    ├── Mod
                    ├── Modpack
                    ├── ResourcePack
                    ├── ShaderPack
                    └── Config
```

每次快照包含运行环境上下文（MC 版本、加载器版本、Java 版本、操作系统）及五类资产的完整状态。各资产共享统一追踪维度：`version`、`addedTime`、`updateTime`、`note`、`isDeleted`。

后端已实现模组维度的核心表结构：

- `mod_info` — 模组全局主数据（名称、来源平台、最新版本）
- `mod_snapshot` — 快照与模组的关联记录（版本、哈希、加载耗时、收藏状态等）

完整五类资产的 Instance/Snapshot 模型已在设计文档与前端 Mock 数据中定义，后端 API 尚在逐步落地。

## 目标用户与典型场景

### 间歇性玩家 — 回归审计

玩家三个月未游玩，回归前通过 Block Snap 查看实例变更全景：哪些模组有新版本、哪些标记了破坏性变更、哪些配置文件可能需要迁移、资源包 Pack Format 是否需同步更新——无需逐一访问多个平台。

### 整合包维护者 — 版本管理

维护者追踪整合包内全部模组的版本偏差，管理配置基线，对比玩家实例的配置漂移，结合平台更新日志辅助制定更新策略。

### 社区协作 — 问题诊断

玩家遇到崩溃或异常时，生成包含完整资产快照的分享链接；管理员打开后可查看资产清单、高亮配置漂移，并对比异常快照与上次正常快照的 Diff，替代传统「发日志 → 人工比对 → 反复沟通」流程。

## 技术架构

### 技术选型

| 层次 | 技术 | 说明 | 状态 |
|------|------|------|------|
| 客户端 | Fabric/Forge Mod (Java 21) | 与 MC 模组生态同语言，可共享数据模型 | 规划中 |
| 后端框架 | Spring Boot 3.3 + Spring Cloud | 微服务治理、网关鉴权、服务发现 | 已落地 |
| 服务注册 | Nacos | 多服务动态发现与配置 | 已落地 |
| API 网关 | Spring Cloud Gateway | 统一入口、JWT 鉴权、路由转发 | 已落地 |
| 关系数据库 | MySQL + MyBatis-Plus | 用户-实例-资产多维关系存储 | 已落地 |
| 缓存 | Redis | 会话管理、Token 校验、热点数据 | 已落地 |
| 对象存储 | 阿里云 OSS | 通用文件上传 | 已落地（暂无调用方） |
| 消息队列 | RabbitMQ | 客户端快照异步上报，削峰填谷 | 规划中，pom.xml 尚无该依赖 |
| 搜索引擎 | Elasticsearch | 全资产全文检索与跨实例聚合 | 规划中，pom.xml 尚无该依赖 |
| 任务调度 | XXL-Job | 平台数据定时采集 | 规划中，pom.xml 尚无该依赖 |
| 前端 | Vite + 原生 JS（目标 Vue 3） | 变更时间线与 Diff 可视化 | 演示阶段 |
| 部署 | Docker Compose | 多中间件环境一致性 | 已落地（本地中间件） |

> 「状态」列的判定标准是代码与依赖清单，不是设计意图——即便某项技术在架构上是必要的，只要 Maven 依赖里还没有它，就标注为「规划中」。这张表会随实现进度更新，是全部文档中优先级最高的一份事实来源。

### 仓库结构

```
block-snap/
├── block-snap-server/          # 后端微服务
│   ├── block-snap-gateway/     # API 网关（JWT 鉴权、路由）
│   ├── block-snap-system/      # 用户系统（注册/登录/账户管理）
│   ├── block-snap-service/     # 业务服务（模组快照查询等）
│   └── block-snap-common/      # 公共模块（JWT、异常处理、常量）
├── block-snap-web/             # Web 前端（演示 UI + Mock 数据）
└── block-snap-文档/             # 设计文档与项目说明
```

| 模块 | 状态 | 说明 |
|------|------|------|
| 用户系统 | 已落地 | 注册、登录、验证码、密码重置、账户绑定/改名三层鉴权（登录态+二次验证）；Gateway JWT + Redis 会话校验 |
| API 网关 | 已落地 | 白名单路由、Token 校验、用户上下文透传 |
| 实例列表 API | 已落地 | `instance` 列表/收藏/备注，含跨库聚合最新快照、模组数、更新数等统计 |
| 模组列表 API | 已落地 | 指定实例最新快照下的 `mod_snapshot` 列表、收藏/备注 |
| 文件上传 API | 已落地（暂未接入业务） | 阿里云 OSS 通用上传接口，当前无调用方（无头像/图标等实际使用场景） |
| Web 前端 | 演示阶段，账户与实例/模组模块已接入真实后端 | 变更时间线、版本更新等页面仍为 Mock 数据 |
| Snapshot 写入/Diff API | **规划中，尚无代码** | 客户端上报接口、快照创建、两次快照 Diff 计算 |
| 客户端 Mod | **规划中，尚无代码** | 五类资产扫描与上报 |
| 消息队列（RabbitMQ） | **规划中，尚无代码与依赖** | 快照异步写入与削峰 |
| 搜索引擎（Elasticsearch） | **规划中，尚无代码与依赖** | 跨实例聚合检索 |
| 平台聚合（CurseForge/Modrinth） | **规划中，尚无代码** | 定时采集、破坏性变更识别 |

> 说明：网关已为 `/svc-snapshot/**` 预留了路由转发规则，但 `block-snap-service` 侧目前只有 `Snapshot` 的数据模型与基础 Mapper，没有对外的 Controller，即该路径当前会转发到一个没有匹配接口的服务，属于「基建先行、接口未跟上」的阶段性状态，而非已可用能力。

前端 v2 数据模型（`Account → Instance → Snapshot → Assets`）已与核心设计对齐；后端正从早期「以启动记录为中心」向「以实例为中心」演进。

- RabbitMQ 快照消费与多资产 Diff 引擎

- Elasticsearch 全文检索与跨实例统计

- 开放 API（供 HMCL、PCL2、Prism Launcher 等第三方启动器集成）
- LLM 辅助更新日志摘要与破坏性变更分类

## 快速开始

### 前端演示

```bash
cd block-snap-web
npm install
npm run dev
```

访问 http://localhost:5173 查看演示页面。用户认证已对接后端 `/sys-user/**` 接口。

### 后端服务

需预先启动 MySQL、Redis、Nacos，分别运行：

- `block-snap-gateway`
- `block-snap-system`（端口 8081）
- `block-snap-service`（端口 8082）

## 项目定位总结

Block Snap 是一个面向 Minecraft 游戏实例的**多资产时序状态追踪与变更知识聚合系统**。它将五种异构游戏资产统一纳入追踪体系，不只记录「现在有什么」，更关注「从何时开始有了什么」以及「时间流逝中各资产间发生了哪些联动变化」。

从工程视角，项目涵盖时序状态追踪、多源异构信息聚合、跨资产关联分析、分布式微服务与高并发异步写入等实践场景。从用户视角，它解决了一个现有 Minecraft 工具链未能有效覆盖的核心问题：

**在时间流逝中，构成游戏实例的五类资产各自经历了怎样的演变，联动变化意味着什么，以及现在应该如何处理这些累积的变更。**
