# Block Snap — Minecraft 实例变更追踪与游戏资产知识聚合平台


## 摘要

Block Snap 是一个面向 Minecraft 玩家与整合包维护者的**实例状态变更追踪系统**。通过在游戏客户端部署轻量级 Fabric/Forge 模组，在每次启动时自动采集当前实例中全部游戏资产的完整快照——涵盖模组、整合包元数据、资源包、光影包、模组配置文件——记录其版本号、文件哈希、文件修改时间戳等标识信息，经由消息队列异步上报至后端服务。服务端对历史快照进行差分对比，并结合 CurseForge、Modrinth 等平台的更新数据进行知识聚合，最终通过 Web 仪表盘向用户呈现**跨时间维度的资产变更全景**：何种资产发生了何种变化、变化的时机与来源、以及变更对实例稳定性的潜在影响。

项目解决的核心问题是：Minecraft 实例由多种异构游戏资产组合而成，这些资产生态更新频繁、信息分散，玩家在中断游玩一段时间后，难以有效获知实例中各资产在此期间发生的变更，以及这些变更对兼容性和游戏体验的综合影响。


## 问题背景

Minecraft Java Edition 围绕游戏实例构建了一个多层次的资产体系，包含以下主要类别：

- **模组（Mods）**：通过 Forge 或 Fabric 加载的 `.jar` 文件，直接扩展或修改游戏机制。更新频繁（热门模组普遍以周甚至日为周期），版本间的 API 变更可能导致不兼容。
- **整合包（Modpacks）**：以 CurseForge/Modrinth 清单或第三方启动器导出文件形式存在的模组集合定义，封装了特定版本的模组组合与配置。
- **资源包（Resource Packs）**：替代或增强游戏纹理、音效、字体、语言的 `.zip` 资源包，许多模组依赖特定版本的资源包以提供完整的视觉体验。
- **光影包（Shader Packs）**：运行于 OptiFine 或 Iris 之上的 GLSL 着色器程序集，高度依赖 GPU 驱动版本与特定模组的渲染管线接口。
- **配置文件（Config Files）**：位于 `config/` 目录下的各模组配置（`.toml`、`.cfg`、`.json`、`.yml` 等），定义了模组的运行参数。配置文件在模组更新后可能需要手动迁移或重置，且常因配置格式变化或键名废弃而导致游戏异常。

这一资产体系存在以下结构性问题：

- **多维更新耦合**：单个模组的版本更新可能联动影响其配置文件格式、依赖的资源包版本以及整合包的整体兼容性声明，各资产类别的变更是相互关联而非孤立的。
- **信息高度碎片化**：模组与整合包的更新日志主要分布在 CurseForge、Modrinth、GitHub Releases；资源包与光影包的更新信息分散在 CurseForge、Planet Minecraft、Modrinth 乃至个人 Discord 社区，缺乏统一的聚合入口。
- **时间隔阂效应**：Minecraft 玩家的游玩模式通常是间歇性的——集中游玩数周后暂停数月，当再次返回时，实例已在时间流逝中发生大量变更，而玩家对各资产变动的信息严重滞后。
- **配置漂移问题**：模组配置文件在玩家使用过程中会被手动调整，但缺乏版本控制，当模组更新且配置格式变更时，玩家难以判断配置是否需要重置或手动迁移。

现有工具（如 Prism Launcher、Modrinth App、CurseForge App）主要服务于"当前状态的资产管理"，即安装、启用、禁用、更新——聚焦于"现在"。它们不持久化记录历史状态，不追踪跨资产类别的关联变更，也不提供跨时间维度的变更认知能力。


## 项目定义

Block Snap 由三个子系统构成：

**客户端采集层**：一个 Fabric/Forge 模组，在游戏加载生命周期中监听启动事件，对实例下的五个资产目录进行全量扫描：

| 资产类别 | 扫描路径 | 采集内容 |
|----------|----------|----------|
| 模组 | `mods/` 目录下所有 `.jar` | 模组 ID、名称、版本（从 `mods.toml` / `fabric.mod.json` 提取）、文件修改时间、SHA-256 哈希、加载耗时 |
| 整合包 | 整合包清单文件 | 整合包名称、版本、所含模组列表及其版本 |
| 资源包 | `resourcepacks/` 目录下所有 `.zip` | 资源包描述、Pack Format 版本（从 `pack.mcmeta` 提取）、文件修改时间、SHA-256 哈希 |
| 光影包 | `shaderpacks/` 目录下所有 `.zip` | 光影包名称（从着色器目录结构推断）、文件修改时间、SHA-256 哈希 |
| 配置文件 | `config/` 目录下所有配置文件（递归） | 文件相对路径、文件修改时间、SHA-256 哈希、关联模组（由文件名命名空间推断） |

所有采集结果聚合为一次完整的实例快照，序列化后发送至 RabbitMQ。采集过程异步执行，不阻塞游戏主线程，对启动时间的影响控制在毫秒级。

**服务端处理层**：基于 Spring Cloud 的微服务集群，包含以下核心服务：
- **快照消费服务**：从 RabbitMQ 消费客户端快照，与用户历史快照进行多维度结构化 Diff——按资产类别分别计算新增、移除、版本变更、文件哈希变更事件，存入关系数据库并索引至 Elasticsearch。
- **平台聚合服务**：通过 XXL-Job 定时调度，从 CurseForge API、Modrinth API 及 GitHub Releases 爬取模组与整合包的版本更新日志及社区反馈；对资源包与光影包平台（CurseForge、Planet Minecraft 等）进行辅助性数据采集；经 NLP 管道提取破坏性变更标记与关键信息摘要。
- **变更分析服务**：将平台聚合数据与用户实例的实际变更进行关联匹配，同时对配置文件变更与模组版本变更进行关联分析（如模组升级后其配置文件是否被自动重置或需要手动迁移），生成个性化的变更时间线与风险评估结果。

**Web 展示层**：基于 Vue 3 的单页应用，按资产类别组织视图，提供变更时间线、各资产类别的 Diff 视图、聚合更新日志阅读、跨资产关联视图（如"某模组更新后，其配置文件与关联资源包的变化"）、风险等级标识、全文搜索、数据导出等功能。通过 WebSocket 接收快照处理完成后的实时推送。


## 资产分类与追踪维度

### 各资产类别的追踪策略

| 资产类别 | 本地追踪 | 平台聚合 | 变更检测方式 |
|----------|----------|----------|--------------|
| 模组 | ✓ 版本 + 哈希 + 时间戳 | CurseForge / Modrinth / GitHub | 版本号差异 + 文件哈希差异 |
| 整合包 | ✓ 版本 + 清单 | CurseForge / Modrinth | 清单内容差异 + 所含模组版本漂移 |
| 资源包 | ✓ 哈希 + 时间戳 + Pack Format | CurseForge / Planet MC（辅助） | 文件哈希差异 + Pack Format 版本差异 |
| 光影包 | ✓ 哈希 + 时间戳 | CurseForge / Modrinth（辅助） | 文件哈希差异 |
| 配置文件 | ✓ 哈希 + 时间戳 + 路径 | 不适用（纯本地资产） | 文件哈希差异 + 关联模组版本差异 |

配置文件是 Block Snap 追踪体系中最具特殊性的资产类别：它们没有外部平台的"版本"概念，但恰恰是玩家手动介入最多的资产。配置文件变更追踪可实现以下场景：
- 当模组从 v1.0 升级至 v2.0 后，系统自动检测该模组的配置文件是否因格式变更而需要重置
- 玩家手动调整某项配置后，可通过配置变更历史回溯修改内容
- 整合包作者可通过共享配置快照确保服务器全体玩家使用一致的配置基线


## 典型使用场景

### 玩家回归后的全资产审计

一位玩家在三个月前安装了包含 150 个模组、8 个资源包、2 个光影包的整合包，随后暂停游玩。当他准备重新进入游戏时，通过 Block Snap 查看实例的变更全景：
- **模组**：23 个模组已发布新版本，其中 2 个标记了破坏性变更，1 个要求更新 Forge 加载器版本。
- **配置文件**：3 个模组的配置文件因版本升级发生格式变化，系统提示可能需要手动迁移或重置。
- **资源包**：配套的 32x 纹理资源包更新了 Pack Format 以兼容新版 Minecraft，需同步更新以保持视觉效果一致。
- **光影包**：1 个光影包版本更新，需配合更新的 Iris 版本使用。
- 每个资产的变更内容已从各平台聚合展示，玩家无需访问多个网站即可全面了解实例在三个月间的演变。

### 整合包维护者的多资产版本管理

一位整合包作者维护着包含数百模组的服务端整合包。通过 Block Snap：
- 系统自动追踪整合包内所有模组的版本状态，对比最新发布版本，生成跨资产偏差报告。
- 配置文件变更历史被纳入版本管理：当作者调整了某项服务端配置后，系统记录该变更，并在分发更新时提醒玩家需要同步更新哪些配置文件。
- 聚合来自各平台的更新日志，辅助判断更新策略。
- 结合所有使用该整合包的玩家实例数据，识别特定模组版本在全服范围内的加载异常与兼容性问题。

### 问题诊断与协作

当玩家遇到启动崩溃或运行异常时，可通过 Block Snap 生成一个包含全部资产快照的可分享诊断链接：
- 接收方（如服务器管理员）打开链接后，可查看该实例的完整资产清单：模组、配置文件、资源包、光影包的版本与哈希信息。
- 系统自动高亮"配置漂移"：哪些配置文件与服务器基线不一致。
- 对比崩溃时间点的快照与上一次正常运行的快照，自动标注变更项，替代传统的"发送日志文件 → 手动比对 → 反复沟通"的流程。


## 数据模型

### 实例快照（Snapshot）

```
Snapshot {
    snapshotId       // 快照唯一标识
    instanceId       // 所属实例
    timestamp        // 采集时间
    runtimeContext {  // 运行环境
        minecraftVersion
        forgeVersion / fabricVersion
        javaVersion
        operatingSystem
    }
    assets: {
        mods:            [ModAsset]
        modpack:         ModpackAsset?
        resourcePacks:   [ResourcePackAsset]
        shaderPacks:     [ShaderPackAsset]
        configs:         [ConfigAsset]
    }
}
```

### 各资产数据结构

**ModAsset（模组）**

| 字段 | 来源 | 说明 |
|------|------|------|
| modId | mods.toml / fabric.mod.json | 全局唯一的模组标识符 |
| name | mods.toml / fabric.mod.json | 模组显示名称 |
| version | mods.toml / fabric.mod.json | 语义化版本号 |
| fileName | 文件系统 | jar 文件名 |
| fileHash | SHA-256 | 文件内容哈希 |
| fileSize | 文件系统 | 文件大小（字节） |
| lastModified | 文件系统 | 文件最后修改时间戳 |
| loadTimeMs | Forge ModLifecycleEvent / Fabric ClientLifecycleEvents | 模组初始化耗时 |
| sourcePlatform | 元数据推断 | CurseForge / Modrinth / GitHub 等 |
| dependencies | mods.toml / fabric.mod.json | 声明的前置依赖模组及版本约束 |

**ModpackAsset（整合包）**

| 字段 | 来源 | 说明 |
|------|------|------|
| name | 整合包清单 | 整合包名称 |
| version | 整合包清单 | 整合包版本号 |
| sourcePlatform | 整合包清单 | 来源平台（CurseForge/Modrinth） |
| manifest | 整合包清单 | 所含模组列表（modId → version 映射） |

**ResourcePackAsset（资源包）**

| 字段 | 来源 | 说明 |
|------|------|------|
| description | pack.mcmeta | 资源包描述文本 |
| packFormat | pack.mcmeta | Pack Format 版本号（决定兼容的 MC 版本） |
| fileName | 文件系统 | zip 文件名 |
| fileHash | SHA-256 | 文件内容哈希 |
| fileSize | 文件系统 | 文件大小 |
| lastModified | 文件系统 | 文件最后修改时间戳 |

**ShaderPackAsset（光影包）**

| 字段 | 来源 | 说明 |
|------|------|------|
| name | 着色器目录结构 | 光影包名称 |
| fileName | 文件系统 | zip 文件名 |
| fileHash | SHA-256 | 文件内容哈希 |
| fileSize | 文件系统 | 文件大小 |
| lastModified | 文件系统 | 文件最后修改时间戳 |
| shaderLoader | 运行时推断 | OptiFine / Iris |

**ConfigAsset（配置文件）**

| 字段 | 来源 | 说明 |
|------|------|------|
| relativePath | 文件系统（相对于 config/ 目录） | 配置文件路径（含子目录） |
| fileHash | SHA-256 | 文件内容哈希 |
| fileSize | 文件系统 | 文件大小 |
| lastModified | 文件系统 | 文件最后修改时间戳 |
| format | 文件扩展名推断 | toml / cfg / json / yml / properties |
| associatedMod | 文件名命名空间推断 | 关联的模组 ID（可空） |

### 平台聚合数据（跨资产通用）

| 字段 | 来源 | 适用资产类别 |
|------|------|--------------|
| 版本发布日志 | CurseForge API / Modrinth API / GitHub Releases | 模组、整合包、资源包、光影包 |
| 版本发布日期 | 同上 | 全部 |
| 下载量 / 评分 | CurseForge / Modrinth | 模组、整合包 |
| 破坏性变更标记 | NLP 管道对更新日志文本分类 | 模组、整合包 |
| 依赖关系图 | 平台 API / 元数据解析 | 模组 |
| 社区反馈摘要 | Reddit / GitHub Issues（可选） | 模组、整合包 |


## 功能规划

### 面向玩家

**变更时间线与审计**
- 以时间轴展示实例历次快照，标注每次启动在各资产类别中捕捉到的增、删、改事件。
- 支持按资产类别（模组/资源包/光影包/配置）筛选视图。

**资产 Diff**
- 任意两次快照之间的结构化差异对比，按资产类别分别呈现：
  - 模组 Diff：新增/移除/版本升降的模组列表
  - 资源包 Diff：文件变更对比（包括 Pack Format 版本变化）
  - 光影包 Diff：新增/移除/更新的光影包
  - 配置 Diff：哪些配置文件内容发生了变化，关联的模组是否也同步更新

**配置漂移分析**
- 标记实例配置与整合包默认配置之间的偏差。
- 当模组升级后检测其配置文件是否被自动重置或遗留旧版本格式，提示玩家是否需要手动迁移。

**聚合更新日志**
- 在统一界面中阅读所有已安装资产的跨平台更新内容，分类为模组更新、资源包更新、光影包更新。
- 按资产类别筛选，按时间或重要性排序。

**风险分级**
- 三色标识系统对资产变更进行兼容性风险判定：
  - 绿色：补丁级更新，无已知兼容性问题
  - 黄色：功能级更新，建议阅读变更说明；或配置文件需要手动复核
  - 红色：破坏性更新，存在已知不兼容或冲突报告；或配置文件格式已变更需要迁移

**全文检索**
- 基于 Elasticsearch 对模组名称、资源包描述、更新日志内容、配置文件名等进行跨类别全文搜索。

**诊断分享**
- 生成实例快照的分享链接，支持选择性地包含或排除特定资产类别（如仅分享模组列表、或包含完整配置快照）。

**加载性能分析**（辅助功能）
- 模组加载耗时排行，以及同一模组不同版本的加载速度对比。
- 资源包与光影包的启用对加载时间的影响评估。

### 面向整合包维护者

**全资产版本追踪**
- 自动对比整合包当前版本与各资产最新版本，按类别生成偏差报告。
- 配置文件基线管理：定义整合包的标准配置基线，对比玩家实例配置偏差。

**自动生成更新说明**
- 基于变更历史与平台聚合数据，自动生成按资产类别组织的终端用户更新说明。

**玩家实例概览**
- 查看使用当前整合包的玩家群体的全资产版本分布、配置漂移情况与加载性能数据。

**兼容性告警**
- 当某模组版本在全服范围内出现大量崩溃报告，或其配置文件格式变更导致玩家配置漂移时，向维护者发出通知。

**统计仪表盘**
- 活跃实例数、各资产类别热门排行、资产更新频率趋势、平均配置漂移度等聚合指标。

### 开放接口

- **REST API**：为第三方启动器（HMCL、PCL2、Prism Launcher 等）提供快照上报、资产查询与配置基线比对接口。
- **数据导出**：支持 JSON、CSV 格式，支持按资产类别选择性导出。
- **Webhook 通知**：支持按资产类别和模组个体配置更新推送策略。


## 与同类工具的核心差异

现有 Minecraft 工具几乎全部聚焦于"当前时刻的资产状态管理"——查看实例中现在有哪些模组、安装或更新它们。它们的视野被限制在"现在"的截面上。

Block Snap 的定位是**跨时间维度的多资产实例变更追踪与知识聚合系统**。它将五种异构资产统一纳入追踪体系，不只记录静态的"有什么"，更关心"从什么时候开始有了什么"以及"在时间流逝中各资产间发生了哪些联动变化"。

| 能力维度 | 传统启动器 | 模组平台客户端 | Block Snap |
|----------|------------|----------------|------------|
| 管理当前模组状态 | ✓ | ✓ | 不涉及 |
| 安装/更新/移除资产 | ✓ | ✓ | 不涉及 |
| 模组历史快照与变更追踪 | — | — | ✓ |
| 资源包变更追踪 | — | — | ✓ |
| 光影包变更追踪 | — | — | ✓ |
| 配置文件变更追踪与漂移分析 | — | — | ✓ |
| 多平台更新日志聚合 | — | 单一平台 | ✓（跨平台） |
| 跨资产关联分析 | — | — | ✓ |
| 兼容性风险自动判定 | — | — | ✓ |
| 实例状态分享与协作诊断 | — | — | ✓ |

## 项目定位

Block Snap 的实质是一个面向 Minecraft 游戏实例的**多资产时序状态追踪与变更知识聚合系统**。它借鉴了 IT 运维领域中 ELK（Elasticsearch + Logstash + Kibana）日志管线的架构思想，但将追踪对象从单一维度的服务端日志扩展为五类异构游戏资产的完整状态快照，将目标场景从故障排查转向玩家侧的实例演变认知与版本风险管理。

从学术视角，该项目涉及以下技术领域：

- **时序状态追踪**：通过周期性全资产快照，构建基于时间轴的多维度实例变更图谱
- **多源异构信息聚合**：从 CurseForge、Modrinth、Planet Minecraft、GitHub 等多个结构差异显著的平台采集并统一结构化游戏资产元数据
- **跨资产关联分析**：在模组版本变更、配置文件漂移、资源包兼容性之间建立关联模型，识别联动变更模式
- **自然语言处理**：对非结构化更新日志文本进行破坏性变更分类与摘要提取
- **分布式微服务工程**：在五类资产、十种以上中间件、高并发写入的复杂场景下实践完整的微服务治理体系

从用户视角，它回答了一个在现有 Minecraft 工具链中未被有效覆盖的问题：**"在时间流逝中，构成我的游戏实例的五类资产各自经历了怎样的演变，它们之间的联动变化意味着什么，以及我现在应该如何处理这些累积的变更。"**
