# Block Snap — 技术选型必要性的"因果链"论证


## 核心思路：不要"列举每个技术有什么用"，要"画因果链"

答辩时被问"为什么用 RabbitMQ"，你如果回答"削峰填谷"——对方会问"你怎么知道有峰？"

正确的防守方式：**每个技术都是前一个技术逼出来的。**


## 完整的因果链

```
Step 1: Minecraft 模组必须用 Java 写
   ↓ （必然）
Step 2: 模组和后端用同一种语言，共享数据模型
   ↓ （自然选择 Java Spring Boot）
Step 3: 后端要存"谁有什么模组" → 需要 MySQL（关系型数据）
   ↓
Step 4: 但数千玩家同时启动游戏 → Tomcat 200 线程扛不住
   ↓ （必然）
Step 5: 需要 RabbitMQ 削峰填谷 → 引入消息队列
   ↓
Step 6: MQ 消费者要写 MySQL + 索引 ES，资源密集
   ↓ （必然）
Step 7: 消费者必须独立部署，不能和 API 服务共用线程池
   ↓ （从单体拆成两个服务）
Step 8: 有了多个服务 → 硬编码 IP 会炸 → 需要 Nacos 服务发现
   ↓ （必然）
Step 9: 多服务 → 前端不能调不同端口 → 需要 Gateway 统一入口 + 鉴权 + 限流
   ↓
Step 10: 想加 AI 分析 changelog（Python + LLM）
   ↓ （必然）
Step 11: Python 和 Java 不同语言 → 必须是独立微服务
   ↓
Step 12: 已有 4+ 服务 → Nacos + Gateway 的价值进一步放大
   ↓
Step 13: 4+ 服务 + 5+ 中间件 → 手动部署不可能 → Docker Compose
```

**每一步都不是"我学了所以要加"，而是"上一步必然导致下一步"。**


## 逐条答辩话术

### Java Spring Boot

> **"为什么非 Java 不可？"**

> Minecraft 模组开发绑定 JVM（Fabric/Forge 只支持 Java/Kotlin）。
> 客户端 Mod 和后端服务使用同一种语言，可以直接共享 POJO（模组数据模型、序列化器）。
> 如果用 Python 写后端，每次模组信息结构一变，Java 和 Python 两边的 Model 要手动对齐——这在生产环境必然出现数据字段不一致。

### MySQL

> **"为什么 MySQL，不是 MongoDB？"**

> 核心数据是账号-实例-模组的多对多关系。一个账号有多个实例，一个实例有多次快照，一次快照有上百个模组。
> 这种关系模型天然适合关系型数据库。MongoDB 的文档模型在这个场景下会导致大量冗余和更新不一致。

### RabbitMQ

> **"为什么需要消息队列？单体 Spring Boot 不能处理吗？"**

> 单体在 100 个用户时能。但 Block Snap 的核心场景是玩家启动游戏时自动上报数据。
> MC 玩家集中在晚间（20:00-22:00）登录。
> 如果集成到 HMCL 启动器（日活几十万），高峰期每秒数千次快照写入请求。
> Tomcat 默认线程池 200，请求堆积 → 超时 → 数据丢失。
> RabbitMQ 的作用是把"同步写入"变成"异步消费"——客户端请求立即返回，由消费者按自己的节奏处理。
> **这和淘宝秒杀系统的架构模式完全一致。**

### Elasticsearch

> **"为什么不用 MySQL 全文索引？"**

> MySQL 的全文索引只支持英文分词，中文模组名完全不可用。
> 核心查询场景——"全服谁在用 Create v0.5.1f"、"Sodium 在各整合包中的版本分布"——
> 需要跨实例、跨快照的聚合分析。MySQL 做这种事是 N+1 查询灾难。
> ES 的倒排索引和聚合查询，让这些操作从秒级降到毫秒级。
> 而且——ES 的数据源是玩家上传的模组信息，不是爬虫。每个用户在启动时上报的模组数据直接索引进 ES。

### Nacos

> **"为什么需要注册中心？单体不够吗？"**

> 当系统拆成实例服务、搜索服务、爬虫服务、AI 服务后，每个服务部署在不同的端口。
> 没有 Nacos，消费者不知道 ES 服务的地址变了——只能硬编码，改一次地址重启一次。
> Nacos 让服务之间通过服务名调用，某个服务扩容/缩容不用改代码。
> 它不是"这个项目需要 Nacos"，而是"微服务架构天然需要服务发现"。

### Spring Cloud Gateway

> **"Nginx 不是也能做网关吗？"**

> Nginx 做反向代理没问题。但 Gateway 的优势在于和 Nacos 原生集成——
> 自动从注册中心拉取服务列表，动态路由，不需要手动改 nginx.conf。
> 加上 Spring Cloud 的限流（Sentinel）、鉴权（Spring Security）都在网关层统一处理，
> 不需要每个微服务各自实现一遍。

### Python + LLM 模块

> **"Java 不能调用大模型吗？"**

> Java 可以调 API。但 LLM 的整个工具链——LangChain、向量数据库、Prompt 模板、
> 模型加载（vLLM/llama.cpp）、RAG 管道——全部是 Python 原生生态。
> 用 Java 硬接 LLM 相当于用汇编写 Web——能做，但意味着放弃整个社区的最佳实践。
> Python 服务独立部署，通过 HTTP 和 Java 通信，这是异构语言协作的标准模式。

### Docker

> **"为什么需要 Docker？"**

> 最终系统包含：Java × 4 服务 + Python × 1 服务 + MySQL + ES + RabbitMQ + Redis + Nacos + Nginx。
> 共 10+ 个组件。在没有 Docker 的情况下：
> - 开发者 A 的 Windows 环境和开发者 B 的 macOS 环境不一致
> - 生产部署需要手动安装 10+ 个软件，配置几十个参数
> - 中间件版本升级可能破坏兼容性
> Docker Compose 一个 `docker-compose up` 解决以上所有问题。


## 答辩最锋利的回答

如果有人质疑"你是不是就是想把学过的都塞进去"：

> "不是我先选技术再找场景。是 Block Snap 这个场景本身就具备这些技术需求的
> 结构特征——高并发用户上传、跨实例全文检索、异构语言 AI 集成、多服务部署。
> 这些是真实存在的工程问题，不是我为了用技术而编的。
>
> 如果只做一个单用户、本地使用的工具，这些技术确实不需要。
> 但 Block Snap 的定位是面向 MC 社区的 SaaS 平台——它的用户量和数据量决定了
> 单体架构在可预见的增长下必然崩溃。我的技术选型是基于这个定位做的容量规划。"


## 总结：必要性 = 因果链，不是清单

```
❌ 错误讲法："我用了 RabbitMQ 因为削峰填谷，用了 ES 因为全文检索..."
   → 听起来像背诵 Checklist

✅ 正确讲法："因为客户端必须是 Java → 后端用了 Java Spring Boot →
              因为要应对高并发 → 引入了 RabbitMQ → 消费者独立成服务 →
              多服务需要 Nacos + Gateway → 要加 AI → Python 独立服务 →
              10+ 组件需要 Docker"
   → 每一步都是上一步逼出来的，不是选的
```
