## 一、项目再定位

现有文档把"产品需求"和"工程能力展示"这两件性质不同的事叙述成同一条平滑的因果链。这是最需要先纠正的认知，因为后面所有必要性论证、流量目标、增长策略都建立在这个认知之上。

Block Snap 实际上是**两层目标的叠加体**：

| 层 | 目标 | 服务对象 |
|---|---|---|
| 表层（产品叙事） | 帮 MC 玩家 / 整合包维护者做「实例资产变更追踪」 | 真实 MC 玩家 |
| 里层（真实驱动） | 用一个真实项目，系统性覆盖后端工程主要场景，作为长期能力证明与技术影响力载体 | 招聘方 / 技术社区 / 未来的自己 |

**一句话定位**：面向 Minecraft 玩家的「实例资产时序状态追踪系统」，同时是一个刻意按企业级后端全栈标准搭建、且已投入真实生产验证的工程能力样板间。

**当前真实状态**（诚实基线，基于对四个后端模块源码、`pom.xml` 依赖清单、网关路由配置的逐项核对，后续所有分析都建立在这上面）：

- **已落地、有真实代码与跨库数据链路支撑**：
  - 用户系统全流程（注册/登录/验证码/密码重置/账户绑定/改名/改昵称），网关三层鉴权（白名单 → JWT → 二次验证 Token）
  - 实例列表 API（`/svc-instance/list`）：跨库聚合当前实例的最新快照、模组数、更新数、收藏/备注
  - 模组列表 API（`/svc-mod/list`）：指定实例最新快照下的模组、收藏/备注
  - 通用文件上传 API（阿里云 OSS），但目前没有任何业务功能调用它（无头像/图标等真实使用场景，是一个已联通但暂未被产品功能消费的能力）
- **配置已就位、但实现是空的**：网关为 `/svc-snapshot/**` 配置了转发路由，但 `block-snap-service` 侧只有 `Snapshot` 的数据模型和空白 `BaseMapper`，没有对外 Controller——这条路径目前实际是"资源已声明，接口未建成"，如果被外部调用会直接落到网关的 404。
- **仅存在于设计文档、`pom.xml` 中零依赖痕迹、尚未落地**：RabbitMQ 消费链路、Elasticsearch 检索、XXL-Job 定时调度、CurseForge/Modrinth 平台聚合、客户端 Mod 本身、LangChain AI 摘要能力

这条基线必须持续维护、如实更新，因为它是本文档第五、第六节所有对外传播策略的前提——一旦对外展示的内容与这条基线脱节，会直接反噬"能力证明"和"GitHub 影响力"这两个目标。

## 二、为什么要做这个项目：必要性的三条独立轴线

不要把业务必要性、架构必要性、能力证明必要性揉成一条链条对外讲述，它们是三条各自独立成立的理由，任何一条单拎出来都要能自洽，不依赖另外两条。

### 轴线 1：业务必要性

MC 长期玩家和整合包维护者存在真实的体感问题：模组更新信息分散在多个平台、间歇性游玩导致回归时信息严重滞后、配置文件缺乏版本追踪。现有工具（Prism Launcher、CurseForge App、Modrinth App）只做"当前时刻"管理，不做历史状态追踪，这是真实的市场空白。

风险提示：这是一个高度依赖第三方生态分发的小众利基市场，产品做得好不等于能自然获得用户，第四节和第六节会展开这一点。

### 轴线 2：架构必要性

正确的表述不是"流量倒逼了微服务"，而是：**这个领域天然存在几条与用户规模无关的边界，提前按边界拆分是在为未来的扩展一次性验证清楚，而不是应对眼下的负载。**

三条天然边界：

1. **认证域与业务域**：变更节奏、故障域、合规要求完全不同，这个拆分在 100 人和 10 万人规模下都应该拆，与规模无关。
2. **写路径与读路径**：客户端高频写入（弱一致性容忍）与 Web 端低频读取（强一致性要求）用同一套线程池和事务边界从设计上就是错的，这是正确性问题，不是性能问题。
3. **同构语言与异构语言**：一旦决定要做 AI 摘要能力，Java 单体在物理上就无法容纳 Python/LangChain 生态，这是唯一一条真正"必然"的边界，不需要论证规模，只需要论证"要不要做这个功能"。

### 轴线 3：能力证明必要性（真实流量诉求的真正意义）

真实用户流量（无论是产品侧的 1k+ 真实使用，还是本文档新增的 GitHub 关注度）**不是用来证明系统扛得住规模**——这个体量任何架构都扛得住——**而是用来证明这不是一个只在本地跑通、只有 Mock 数据、只在文档里好看的 Demo 项目**。

这是真实性证据，不是规模证据。这个区分决定了后续所有增长动作该往哪个方向投入：不是压测报告和扩容演练，而是可验证的真实用户行为证据（自然的时间分布、真实的设备指纹分布、真实的留存曲线、真实的外部关注度）。

## 三、技术栈必要性：模块化拆解

每个技术标注独立的必要性来源和当前验证状态，任何一行都可以单独被追问、单独回答，不依赖其他行是否成立。

| 技术 | 必要性来源（可独立成立） | 当前状态 |
|---|---|---|
| Java | ①MC 模组生态强制 JVM，客户端服务端共享数据模型 ②国内后端招聘市场主流语言，能力展示需在被验证的主流赛道上竞争 ③Spring 生态在"微服务全套组件"上工具链最完整，架构展示边际成本最低 | 已验证，全部服务落地 |
| 微服务拆分 | ①认证域与业务域边界天然存在 ②独立数据库、逻辑关联，拆分是数据模型决定的 ③微服务治理本身是考察点，需要真实可演示 | 已验证，三服务在跑，网关鉴权链路完整 |
| Nacos / Gateway | 多服务后必然需要服务发现与统一入口，避免硬编码 | 已验证 |
| MySQL / Redis | 强关系型数据 + 临时态会话/验证码数据的自然选型 | 已验证，全链路有真实查询 |
| RabbitMQ | 客户端上报与服务端处理解耦，应对高峰写入 | **未验证**，无消费者实现，是目标架构假设 |
| Elasticsearch | 跨实例跨快照的聚合查询，中文分词场景 MySQL 不适用 | **未验证**，无索引写入链路 |
| Python + LangChain | LLM 工具链（RAG、Prompt、向量库）是 Python 原生生态 | **未启动**，仅在远期路线图 |
| Docker Compose | 组件数量导致的环境一致性问题，与流量无关 | 已用于本地中间件环境 |

对外沟通原则：凡是"未验证"状态的技术，只能说"目标架构中的必要环节，当前处于规划阶段"，不能表述为已兑现的能力。这是可信度的底线，尤其在下一节的 GitHub 公开策略下会被技术受众直接检验。

## 四、可行性风险清单

| 风险 | 描述 | 应对方向 |
|---|---|---|
| 范围风险 | 客户端 Mod、五类资产建模、微服务全套治理、AI 能力、前端可视化——单人推进的工作量清单接近中型团队规模 | 严格阶段优先级，不同时铺开，先打通 Instance/Snapshot 全链路，MQ/ES/AI 依次跟进 |
| 获客风险 | 1k+ 真实用户的达成路径与后端架构完全无关，是最难的部分 | 需要真实分发渠道：社区口碑、整合包作者背书、启动器集成，见第六节 |
| 时机风险 | 架构搭建速度超前于获客速度，会出现"重装甲打不到苍蝇"，负载证据薄弱反而削弱必要性论证 | 现阶段必要性论证依赖轴线 2、3（边界与能力证明），不依赖轴线 1 的规模倒逼说辞 |
| 证据链风险 | "1k+ 真实流量"若无法举证会被质疑为刷量或虚构 | 提前设计可验证证据：设备指纹自然分布、访问时间符合真实作息、留存曲线、非人为刷量特征 |

## 五、新增分支目标：GitHub Star 增长策略分析

用户提出的新分支目标：在 GitHub 上获取数百 Star，以此获取用户使用/访问流量。这个目标需要独立分析，因为它和"1k+ 真实产品流量"这个目标**服务的是不同的受众，走的是不同的转化路径，甚至在执行层面可能互相打架**。

### 5.1 核心判断：Star 和真实产品流量是两个几乎不重叠的漏斗

GitHub 的访问群体本质是**开发者**，而 Block Snap 的目标用户是 **MC 玩家 / 整合包维护者**，这两个群体的交集很小：

- 给项目点 Star 的人，多数是因为对架构、代码质量、工程实践感兴趣，未必玩 Minecraft，更未必会真的注册使用这个产品
- 真正会长期使用这个产品的 MC 玩家，多数不会主动逛 GitHub 找工具，他们的信息获取路径是 Discord 社区、模组论坛、B 站/YouTube 攻略视频、整合包作者推荐

**结论：GitHub Star 数量本质上服务的是第二节"轴线 3：能力证明"，而非"轴线 1：业务必要性"。** 把它当作产品获客渠道会导致打法失焦——为了讨好开发者做的技术炫技向内容（架构图、微服务治理细节），对 MC 玩家没有意义；为了讨好 MC 玩家做的产品化包装（游戏截图、使用教程），对开发者社区缺乏技术深度，同样拿不到 Star。

**这不代表两个目标不能同时追求，而是需要用两套完全不同的内容和渠道分别触达，不能指望一个动作同时喂饱两个漏斗。**

### 5.2 两套受众的分别打法

**面向开发者（Star 增长）：**

- 仓库首页（README）是唯一的转化入口，需要在几秒内传达"这是一个真实运行的复杂系统，不是玩具项目"：架构图、真实可点击的在线 Demo 链接、关键设计取舍的简要说明（呼应第三节的诚实标注）
- 内容营销比仓库本身更重要：围绕具体工程难点写技术文章（例如"如何在无物理外键的跨库场景下设计用户标记表""网关三层鉴权模型的取舍"），发布在技术社区，反向为仓库导流，这类内容天然适合被转发、收藏、引用
- 渠道：技术社区、编程类社区、相关技术生态的资源汇总列表，都是被验证过的 Star 增长路径
- 时机：应在核心链路（Instance/Snapshot 全链路、MQ、AI 摘要）都补齐到"经得起代码审视"的程度后再做集中式的公开传播，避免"文档写得多、代码跟不上"被技术受众发现后反向扣分

**面向 MC 玩家（真实产品流量）：**

- 完全不提架构，只讲玩家能感知的价值："再也不会漏掉模组更新""回归游戏前先看这里"
- 渠道：MC 相关社区、Discord 社群、内容创作者合作、整合包维护者的直接背书——这条路径的核心资源是"关系"，不是"内容质量"
- 这条路径的速度天然比 Star 增长慢，且高度依赖能否拿到几个有话语权的社区节点（知名整合包作者、活跃社群管理员）的认可

### 5.3 公开前的尽调清单（本次分析中发现的真实风险项）

一旦决定把仓库公开，代码和文档会直接暴露在技术受众的审视下，这不只是"看起来专业"的问题，也是实际的安全和信任问题。以下是本次分析过程中确认存在的具体风险，需要在公开前逐项清零：

| 风险项 | 具体情况 | 处理建议 |
|---|---|---|
| **凭证明文暴露（已修复两处，仍需轮换密钥）** | 本次代码审查中发现同一对阿里云 AccessKey/Secret 明文出现在两处：`block-snap-文档/key.txt`（已删除）与 `block-snap-common/src/main/resources/application.yml`（已改为环境变量占位符）。两处都曾是 Git 未跟踪状态，尚未进入过提交历史，但已经在本次对话中以文本形式暴露过 | 文件层面的问题已处理；**密钥本身仍应视为已泄露，必须在阿里云控制台禁用/轮换**，这一步是人为操作，无法通过改代码替代。仓库公开前对全部提交历史做一次凭证扫描，确认没有历史提交中残留过密钥 |
| **文档与代码进度不一致** | 现有《项目简介》等文档对 MQ、ES、平台聚合等模块的表述容易让读者误以为已经落地，实际是规划中 | 公开前统一按第一节"诚实基线"表述，明确标注"已实现/规划中"，避免被技术受众发现后质疑项目诚信度 |
| **默认弱凭证** | 本地开发环境的数据库等默认账号密码若与仓库中的示例配置一致，一旦被直接照搬到生产环境会有风险 | 仓库中的示例配置一律使用明显的占位符，并在 README 中提示生产环境必须自行替换 |
| **开源范围未决策** | 尚未决定是否连认证/账户体系（安全敏感度最高的部分）也一并开源 | 这是需要创始人明确拍板的战略决策，而非技术细节：全部开源有利于赢得开发者信任与 Star（"敢把安全代码给你看"本身是加分项），但会提高代码质量审视门槛；只开源客户端 Mod 与部分业务模块则风险更低，但故事完整性和吸引力会打折扣 |
| **License 选择未决策** | 尚未选定开源协议 | 需结合是否保留未来商业化空间来决定，这也是创始人需要拍板的问题，不是纯技术问题 |

### 5.4 对"Star 数量目标"的预期管理

数百 Star 对于一个有真实开源价值、且执行到位（好的 README、好的发布时机、持续的技术内容输出）的项目是现实可达成的目标，但不应被当作确定性结果，也不应该和"真实产品流量"目标共用同一个时间表——前者可能在几周到几个月内因为一次好的发布节奏而快速积累，后者大概率是一个更缓慢、依赖真实社区关系积累的过程。建议把两个目标分别设定独立的时间表和成功判定标准，避免因为其中一个目标进展慢而误判整体项目进度。

## 六、文档模块化落地情况

呼应本次分析开始时指出的问题——现有文档把多个独立问题揉在同一篇叙事里。以下是基于对四个后端模块源码逐项核对后，本轮已执行的整理结果，以及仍然建议但尚未做的后续动作。

**本轮已执行：**

- **删除** `Block-Snap-项目简介.md`（旧版）：与《项目简介-正式版》高度重叠，且旧版把五类资产（模组/整合包/资源包/光影包/配置文件）的字段结构写得像已实现的定案设计，而实际只有"模组"这一类有代码支撑，容易误导读者，保留正式版即可。
- **删除** `key.txt`：明文密钥文件，安全风险，见第五节尽调清单。
- **修正**《项目简介-正式版》：把"当前进展"表拆得更细——区分"实例/模组列表 API 已落地"和"快照写入/Diff API 规划中"（此前笼统写作"Instance/Snapshot 全链路 API 规划中"，但实例与模组列表其实已经是真实可用的接口）；技术选型表新增"状态"列，逐项标注已落地/规划中，并附上"以 `pom.xml` 依赖为判定标准"的说明。
- **修正**《数据库表结构与关联关系》：4.1 节原文说模组列表查询对 `mod_info` 是 LEFT JOIN，实际读取 `ModSnapshotMapper.xml` 后确认是 INNER JOIN，已修正并补充了这个差异可能导致的行为（`mod_info_id` 为空的模组会整行消失）。
- **修正**《sys-user接口与用户数据流转》：补充了原文档遗漏的三个真实存在的接口（`getAccount`、`update-nickname`、`update-username`），并修正了二次验证拦截范围的描述（实际拦截 3 个路径，原文档只写了 2 个）。
- **标注**《技术选型答辩》：加了顶部使用说明，明确这是话术训练材料而非现状说明；修正了 Elasticsearch 段落里一句容易被读成"已经在用"的表述。

**仍然建议、本轮未做（工作量较大，需要单独安排）：**

- 把《技术选型答辩》按中间件拆成多份简短的技术决策记录（每份只回答"解决了什么问题、验证到什么程度、去掉会怎样"），目前仍是一整篇因果链叙事，拆分本身不改变准确性问题，纯粹是可维护性优化，优先级低于本轮的事实修正。
- 《数据库表结构与关联关系》《sys-user接口与用户数据流转》属于实现层文档，本轮只做了具体错误的修正，保持原有结构不变。
